## A. Warm-ups

**A1. FALSE.** TCP handles *message loss*, not *failure in general*. A server
crash still loses in-flight calls and any server-side state; the client still
needs timeouts to detect an unresponsive peer, and the application still needs
semantics (at-most-once?) for calls that were in flight. TCP also cannot tell
"slow" from "dead". Reliability of the pipe is not reliability of the service.

**A2. FALSE.** UDP includes a **checksum**, so it detects some corruption. What
it never does is *recover* anything: loss goes unreported and unrepaired. It
guards weakly against corruption and not at all against loss.

**A3. FALSE.** The scheme gives exactly-once **only in the absence of total
failure**. If the server crashes after executing a request but before its
reply escapes, the client cannot distinguish "executed, reply lost" from
"never arrived": retrying risks re-execution, not retrying risks zero
executions. What the machinery genuinely provides across failure is
**at-most-once** (with duplicate suppression) — see B1(c).

**A4. FALSE**, twice over. First, the ack is generated by the *communication
layer* on receipt, saying nothing about the application above it. Second, the
protocol deliberately acks even **duplicates** while suppressing their
delivery — so an ack is provably compatible with "the application never saw
this copy".

**A5. FALSE.** Performance was a problem (programmers had to organise
computation to avoid communication, defeating the point), but the chapter is
explicit that the *largest* problem is **failure**: a crashed machine takes a
region of the shared address space with it, and no reasonable semantics exists
for a linked list whose `next` pointer refers to vanished memory.

**A6. FALSE.** The goal is to make it *feel* similar, but the differences are
structural: latency is orders of magnitude larger; new failure modes exist
(partial failure, lost replies) that a local call cannot exhibit; and
by-reference arguments don't survive the trip — a pointer is meaningless in
another address space, which is why stubs must marshal data by value or by
annotation.

**A7. FALSE.** It says lower-level reliability can never be **sufficient** —
the guarantee must be established end-to-end, at the application. The
corollary explicitly permits, and often recommends, reliability below as a
**performance optimisation**. See B4(c).

**A8. FALSE.** If T < RTT, the timer expires before *any* reply could
possibly arrive, so every single call retransmits needlessly — wasting sender
CPU and network resources (and, under load, aggravating the very losses it
reacts to). Correct tuning waits "just long enough to detect packet loss but no
longer", i.e. slightly above the expected RTT.

## B. Protocol arithmetic

**B1.**
**(a)**

Following ch. 48, the sender stamps each message with its current counter and
increments **immediately after sending** (the message carries the pre-increment
value); the receiver delivers only when the incoming ID equals its own counter,
then increments.

| Event | S counter | R counter | Delivered to app? |
|---|---|---|---|
| S sends msg #1 (ID 1), counter→2 | 2 | 1 | — |
| R receives #1: ID 1 = counter 1 → ack, deliver, counter→2 | 2 | 2 | **yes** |
| S receives ack #1 | 2 | 2 | — |
| S sends msg #2 (ID 2), counter→3 | 3 | 2 | — |
| R receives #2: ID 2 = counter 2 → ack (lost), deliver, counter→3 | 3 | 3 | **yes** |
| S times out, resends #2 (ID 2) | 3 | 3 | — |
| R receives #2 again: ID 2 < counter 3 → **ack, suppress** | 3 | 3 | no — duplicate |
| S receives ack #2 | 3 | 3 | — |
| S sends msg #3 (ID 3, lost), counter→4 | 4 | 3 | — |
| S times out, resends #3 (ID 3) | 4 | 3 | — |
| R receives #3: ID 3 = counter 3 → ack, deliver, counter→4 | 4 | 4 | **yes** |
| S receives ack #3 | 4 | 4 | — |

Three messages sent, five transmissions, each delivered exactly once. The
crucial row is the duplicate of #2: **acked but not delivered**.

**(b)** Remembering every ID ever seen needs **unbounded memory** at the
receiver; the counter needs O(1) state. The price: the counter assumes an
**ordered channel from a single sender** — it can only recognise "the next
expected message", so it presumes messages are not reordered past one another
and that there is one counter per sender/connection. The remember-everything
scheme tolerates arbitrary reordering and interleaving.

**(c)** R reboots with its counter back at 1 (or any initial value). S's next
message carries ID N; R expects 1. Depending on the comparison logic, R either
discards live messages as "duplicates" (N treated as already seen) or —
worse — re-accepts replayed old messages as new. Either way the invariant is
gone. Lesson: the "reliable channel" is really **shared state distributed
across both endpoints** — exactly the kind of distributed state NFS (next
week) is designed to avoid holding. Connection re-establishment with fresh
agreed counters is the standard repair.

*Marking note: (c) wants the word "state" — the scheme is stateful, and crash
recovery of protocol state is the whole game. Answers that only say "it
breaks" earn little.*

**B2.**
**(a)** q = (1 − 0.1)² = **0.81**. Expected attempts = 1/q ≈ **1.23**.
**(b)** Every failed attempt costs the full timeout T; the final successful
attempt costs one RTT = 10 ms:

E[latency] = RTT + (1/q − 1)·T = 10 + 0.235 × 40 ≈ **19.4 ms**.

(Accept small variations in modelling — e.g. counting server time — if the
structure "successes cost RTT, failures cost T" is right.)
**(c)** With T = 8 ms < RTT = 10 ms the timer *always* fires before the reply
can arrive: every exchange sends at least one spurious retransmission —
roughly **doubling request traffic** — even with zero loss. Correctness
survives because the receiver's sequence counter acks-and-suppresses the
duplicate (B1). The cost is pure waste: sender CPU, network capacity, and
receiver work, precisely what the chapter's timeout tip warns about.
**(d)** Waiting time in timeouts: 40 + 80 + 160 + 320 + 640 = **1240 ms**
(~1.24 s). Loss at a busy server often *means overload* — its buffers are
full. Backing off exponentially sheds offered load so the server can drain;
fixed-interval retry keeps the aggregate retry rate constant (or growing, as
more clients join the retrying crowd), sustaining the overload it is reacting
to. The chapter credits the scheme to the Aloha network, later adopted by
Ethernet.

**B3.**
**(a)** Reliable-transport style: request, ack, reply, ack = **4 packets**.
RPC-layer style: request, reply-as-ack = **2 packets** (the reply
acknowledges the request; the reply's own ack rides on the client's next
request — and if no next call comes, a retransmission *solicits* an explicit
ack, which is Birrell & Nelson's actual mechanism: they considered and
rejected sending spontaneous acks). Saving: **2 packets per call** — the
redundant acks that a request/response pattern makes unnecessary.
**(b)** 20,000 calls/s × 4 = 80,000 pkt/s versus × 2 = 40,000 pkt/s. The
difference, 40,000 pkt/s × 10 µs = 0.4 s of CPU per second = **40% of one
core** spent on acknowledgment overhead alone.
**(c)** *(i) Large arguments:* once a call spans many packets you need
fragmentation, reassembly, flow control and congestion control — at which
point you are re-implementing TCP, probably worse; ch. 48 notes the RPC
run-time must supply exactly this if the lower layer doesn't. *(ii) Few,
long-lived, high-volume connections* (or wide-area paths where congestion
fairness matters): connection setup and per-connection state amortise away,
and TCP's mature congestion machinery is an asset, not overhead. The 2-packet
saving matters most for many clients making small, sporadic calls.

**B4.**
**(a)** From the Aside: (1) the bytes are corrupted **in the sender's
memory** before the transport ever sees them; (2) the receiver **writes them
to disk wrongly** (or the disk corrupts them) after the transport has
delivered them faithfully. In both cases every byte was "reliably delivered"
and the file is still wrong.
**(b)** After the entire transfer completes, **read the file back from B's
disk**, compute a checksum over it, and compare with a checksum computed over
the file **on A's disk**. The check must span the true endpoints — disk to
disk — and run after the last step that could corrupt data (hence read-back,
not checksum-of-what-we-sent).
**(c)** Take ~10 KB packets, so 10 GB is ~10⁶ packets (at the standard 1.5 KB
MTU it is ~6.7 × 10⁶ — the argument only sharpens); at a 10⁻⁵ per-packet
corruption rate, ~10 corruptions are *expected* (≈67 at the smaller packet
size). Without link-level recovery, any failure of the end-to-end check forces
a **whole-file retransfer** — and the probability that the whole file passes in
one go, (1 − 10⁻⁵)^N, is essentially zero (~4.5 × 10⁻⁵ at N = 10⁶), so
end-to-end-only never terminates in practice. With link-level checksums and retransmission,
each bad packet is repaired locally at the cost of one packet, and the
end-to-end check almost always passes first time. That is the corollary:
lower-level reliability buys no guarantee but enormous efficiency.

## C. Discussion and design critique

**C1.** The property is an **explicit, boundable failure interface**. A
procedure call has clear seams: it starts, it ends, and a distribution layer
can surface failure at those seams (timeout, exception, "call failed —
at-most-once") for the programmer to handle per call. A load instruction has
no such seam: memory access is assumed universal, instantaneous and
infallible by every line of code ever written against it, so when a remote
page vanishes there is *no program point* at which failure can sensibly be
reported — the chapter's dangling-linked-list image. Abstractions survive
distribution when their contract can absorb partial failure; DSM's contract
could not, and no implementation cleverness fixes a contract. (Strong answers
may add the performance-transparency point: calls are visibly expensive,
loads are assumed cheap, so RPC's cost model is honest where DSM's lies.)

**C2.** Their argument: RPC traffic is dominated by **small, frequent,
latency-sensitive request/response pairs**. A bulk-transfer protocol pays
per-connection setup/teardown, window management and acknowledgment traffic
that are all designed to amortise over long streams; for a single small call
nothing amortises, so the overheads dominate end-to-end latency (B3's
redundant-ack count is the simplest instance). A protocol specialised to
request/response — reply doubles as ack, minimal state per call — strips all
of it. The choice flips when calls stop being small and sporadic: **large
transfers** (you need the streaming machinery anyway) or **sustained
high-volume flows over shared wide-area paths** (you need congestion control
that is fair to other traffic). Modern examples: on the specialised side,
DNS-over-UDP or memcached-style datacenter request/response; on the other,
gRPC deliberately rides on TCP (HTTP/2) because its calls traverse the open
internet, multiplex many streams, and carry arbitrary payload sizes.

**C3.** A strong answer is organised around the three stated constraints.

**Server-side state.** Design T holds a TCP control block per client —
buffers, windows, timers; at 10,000 clients that is real memory (order
tens–hundreds of MB with buffers), plus kernel timer and keepalive work, all
resident whether or not a client is calling. Design U holds only the
duplicate cache and per-client counters — tens of bytes each — and, being
RPC-layer state, *it* can be sized, aged and shed deliberately.

**Crash semantics — the decisive point.** TCP's guarantees live inside one
connection and **die with it**. After a server reboot every connection resets;
the client learns only that the pipe broke, not whether the in-flight call
executed. Design T therefore does **not** provide at-most-once across a
crash — the constraint it was trusted to satisfy — unless an RPC layer with
IDs and a duplicate cache is added back on top, which is Design U wearing a
TCP costume. Design U at least owns the machinery (per-client IDs, duplicate
cache) that a recovery discipline can build on — though note a subtlety the
best answers catch: a duplicate cache held in server memory is *also* lost at
reboot, so genuine at-most-once across crashes needs the cache made
persistent, or a reboot epoch in the IDs, or idempotent operations (NFS's
route, next week).

**Overload.** Design U with exponential back-off (B2d) sheds load and
degrades gracefully. Design T adds TCP's own retransmissions and keepalives
per connection, plus SYN storms as 10,000 clients reconnect after any blip —
reconnection stampedes are exactly the failure mode a stateless datagram
design avoids.

**Verdict.** Under these constraints, Design U — the constraint that matters
(at-most-once across reboot) is unmet by T and merely incomplete in U, and U's
state and overload story is better at this fan-in. **What would flip it:** a
measurement of argument/result sizes. If tracing shows calls routinely
exceeding a few KB (multi-packet, needing fragmentation, ordering and flow
control), Design U is on the road to reimplementing TCP badly, and T-plus-an-
RPC-layer becomes the honest choice. Numbers about state cost or reconnect
storms could tilt it too, but payload size is the classic decider.

*Marking note: the trap in this question is treating "TCP is reliable" as
addressing crash semantics at all. Answers that spot the distinction —
transport reliability vs call semantics across failure — and notice that the
duplicate cache itself is volatile state, are at the top of the band.*
